§ 01The master pipeline

Every successful interaction passes through seven stages. Each stage has a defined actor (user, device, or cloud), a defined input, and a defined output. The contract between stages is what makes the system testable and the precision claims measurable.

STAGE01

Sketch capture

ACTOR · USER + DEVICE

The user opens the app, selects a domain, and draws on the canvas. Every stroke is recorded as a sequence of points with timestamps and (where available) pressure. Voice annotation runs in parallel as an optional second channel.

INPUT
Touch events · stylus events · microphone stream · domain selection
OUTPUT
Stroke[] · audio transcript · domain tag · rasterized PNG of canvas
STAGE02

Primitive detection

ACTOR · DEVICE (on-device, no network)

OpenCV runs over the canvas image to identify geometric primitives — rectangles, circles, ellipses, lines, arcs — and their bounding boxes, centers, and proportions. This pass is deterministic and fast (~150ms on a mid-range phone).

INPUT
Rasterized PNG · stroke order metadata
OUTPUT
Primitive[] — [{type, bbox, params}] · spatial relations
STAGE03

Semantic interpretation

ACTOR · CLOUD AI (Claude Sonnet or equivalent)

A multimodal model receives the sketch image, the primitives list, the voice transcript, and the domain tag. It returns a structured assembly specification through forced tool-call output — no free text, only validated function invocations against the closed vocabulary.

INPUT
PNG · primitives[] · transcript · domain · vocabulary
OUTPUT
AssemblySpec{parts[], relations[], confidence, ambiguities[]}
STAGE04

Validation overlay

ACTOR · USER + DEVICE

The device renders a clean schematic from the AssemblySpec and overlays it semi-transparent on the user's original sketch. The user inspects, taps to correct any misidentified part, and confirms. Corrections re-enter the spec without another AI call.

INPUT
AssemblySpec · user's original PNG · tap events
OUTPUT
ConfirmedSpec — user-validated assembly description
STAGE05

Refinement dialog

ACTOR · USER + DEVICE

The system asks contextual follow-up questions tailored to the confirmed parts: material, dimensions (pre-filled from sketch proportions), context, render style. Each is a tap-to-select chip set. Voice answers are also accepted.

INPUT
ConfirmedSpec · tap or voice selections
OUTPUT
EnrichedSpec — full description ready for rendering
STAGE06

Rendering

ACTOR · CLOUD (AI path) or DEVICE (procedural path)

Two rendering paths run from the same spec. The procedural path builds a parametric 3D model deterministically. The AI path generates a photoreal image using ControlNet conditioned on the original sketch so geometry is preserved. Users see whichever they selected first; the other is available with one tap.

INPUT
EnrichedSpec · original sketch (for AI control conditioning)
OUTPUT
Render — GLB 3D model OR PNG photoreal image
STAGE07

Iterate or export

ACTOR · USER

The user accepts the render and exports, or requests adjustments in natural language ("thicker pipe," "industrial setting," "brushed metal instead"). Adjustments update the EnrichedSpec and re-trigger rendering. The geometric lock from the original sketch is preserved across iterations.

INPUT
Render · user feedback (text or option chips)
OUTPUT
Final asset (PNG / SVG / GLB / JSON) · project saved to history

§ 02Critical decision points

The pipeline branches at three places. Each is a moment where the system must choose what to do based on confidence or user input. These are also where most failures occur, so they deserve explicit design.

Decision point A — After interpretation

How confident is the AI interpretation?
confidence >= 0.85  →  Show overlay with confirmation prompt only
0.60 <= confidence < 0.85  →  Show overlay AND prompt for material/dimension early
confidence < 0.60  →  Show ambiguity dialog: "I'm seeing either X or Y — which is it?"
interpretation failed  →  Offer manual primitive labeling fallback (user assigns part types directly)

Decision point B — Render path selection

Which rendering path runs first?
style = blueprint OR CAD  →  Procedural path only (faster, exact)
style = photoreal OR clay  →  AI path with ControlNet sketch conditioning
user has Pro tier  →  Both paths run in parallel, user toggles freely
offline / no network  →  Procedural path forced, AI path queued for next online session

Decision point C — After render delivery

What does the user do next?
accepts as-is  →  Export menu opens, project saves automatically
requests adjustment  →  Spec updates, re-render with same sketch lock
rejects entirely  →  Return to refinement stage, all chips re-selectable
starts over  →  Old project saved to history, fresh canvas opens

§ 03Sub-flows

Three flows that branch off the main pipeline and deserve their own attention.

Voice annotation flow

SUB-FLOW · A
  1. User holds the microphone button on the canvas. Visual feedback shows a live waveform and "listening…" indicator.
  2. Speech is transcribed locally using the device's native speech-to-text (iOS Speech framework / Android SpeechRecognizer) — no audio leaves the device.
  3. Transcript is timestamped against the strokes being drawn at that moment. {stroke_id: 3, transcript: "this is a steel cylinder, 1.5 meters tall"}
  4. On submission, the bound transcripts are sent alongside the sketch to the interpretation model as explicit hints: "User said about region X: ..."
  5. The model uses the hints to disambiguate competing interpretations and to pre-populate material and dimension fields in the refinement stage.

Correction flow (user fixes an interpretation error)

SUB-FLOW · B
  1. User sees the schematic overlay on stage 4 and notices a part is labeled incorrectly — "this is a flange, not a cap."
  2. User taps the part. A bottom sheet opens listing alternative part types from the active domain's vocabulary.
  3. User selects the correct type. The local AssemblySpec updates immediately — no new AI call required.
  4. The schematic re-renders to reflect the correction. Geometry is preserved; only the semantic label changes.
  5. The correction is logged anonymously (with user consent) as training signal: "On sketch X, the model said cap, the user said flange." This feeds future fine-tuning.

Iteration flow (natural-language adjustment)

SUB-FLOW · C
  1. User sees the first render and types or speaks an adjustment: "make the pipe thicker and change to brushed steel."
  2. A lightweight AI call translates the natural-language adjustment into structured edits to the EnrichedSpec — {part: "pipe_1", diameter: ×1.4, material: "brushed_steel"}.
  3. The edit is shown to the user as a confirmation chip before applying: "I'll thicken the pipe and switch to brushed steel — yes?"
  4. On confirmation, the spec updates and re-rendering is queued. The original sketch is reused as the ControlNet conditioning image so the geometric lock holds.
  5. Previous renders remain accessible from a version slider — users can A/B between iterations without losing prior work.

§ 04The session state machine

From the app's perspective, every session is a state machine. Knowing which state we're in determines what UI is visible, what the user can do, and what the system is processing in the background.

  [IDLE]
       │
       │  user taps "new sketch"
       ▼
  [CANVAS_ACTIVE]  ────────────────┐
       │                                  │
       │  user taps "interpret"             │ undo / clear
       ▼                                  │
  [DETECTING_PRIMITIVES]  ───────────┘
       │
       │  primitives ready (~150ms)
       ▼
  [INTERPRETING]
       │
       │  spec returned (~2-4s)
       ▼
  [AWAITING_CONFIRMATION]  ─────────┐
       │                                  │
       │  user confirms                    │ user corrects
       ▼                                  ▼
  [REFINING]                    [AWAITING_CONFIRMATION]
       │                                  (loops)
       │  all chips selected
       ▼
  [GENERATING]  ──────────────────┐
       │                                  │
       │  render ready (~5-15s)             │ generation error
       ▼                                  ▼
  [RESULT]                       [ERROR_RECOVERY]
       │
       │  user accepts ──▶ [EXPORTED] ──▶ [IDLE]
       │  user iterates ──▶ [REFINING]  (with spec preserved)
       │  user starts over ──▶ [IDLE]

§ 05Walkthrough — a real session

To make the abstract concrete, here is a complete session from a plumber sketching a fitting for a customer. The example traces through every stage above.

SCENARIO · DOMAIN: PLUMBING

Plumber sketching a tee fitting with a reducer for a client quote.

T+0s
Plumber opens SketchForge on her phone in the customer's basement. She taps Plumbing as the domain.
T+5s
She draws a horizontal pipe, then a vertical pipe branching upward from its center, then a small circle on the right end of the horizontal pipe to indicate a reducer. While drawing, she holds the mic and says: "three-quarter inch tee, reducing to half inch on the right."
T+18s
She taps Interpret. The app shows a brief "Reading your sketch…" state. Primitive detection runs locally; the AI call runs in the cloud.
T+21s
A schematic overlay appears on her sketch: a clean tee fitting with a reducer. Each part is labeled. The app says: "Tee fitting with a 3/4 inch main run and a 1/2 inch reducer on the right outlet. 92% confident. Tap any part to correct."
T+24s
She taps the central vertical pipe — it was labeled as a branch but she meant it as the inlet. The bottom sheet opens; she taps "Inlet". The schematic re-labels instantly.
T+28s
She taps Confirm. The refinement chips appear: Material: Copper · PVC · PEX · Brass  |  Context: Residential · Industrial  |  Style: Photoreal · Schematic · Blueprint
T+32s
She accepts the defaults (the voice transcript pre-selected Copper, and Residential was the default for plumbing). She taps Generate.
T+41s
A photorealistic copper tee fitting with the right-side reducer appears, geometry matching her sketch exactly. She also sees a small toggle: 3D model — tapping it shows the same fitting as a rotatable procedural render with exact dimensions labeled.
T+50s
She taps Export → PNG and texts it to her customer with a price quote. Total time from opening the app to sent message: under a minute.

§ 06Failure modes and recovery

Three things go wrong often enough to design for explicitly.

FAILURE 01 · LOW INTERPRETATION CONFIDENCE

What happens: The model returns confidence below 0.6 — the sketch is ambiguous or the shapes don't match the domain vocabulary. Recovery: Show the ambiguity dialog with the top two interpretations side-by-side. If the user rejects both, offer manual primitive labeling — the user assigns part types directly and skips the AI interpretation step entirely.

FAILURE 02 · GENERATION TIMEOUT OR ERROR

What happens: The AI generation call exceeds 30 seconds or returns an error. Recovery: Fall back to procedural rendering automatically — geometry is always exact, even if photorealism is unavailable. Surface a non-blocking toast: "Showing 3D model instead. We'll try the photoreal version when you're back online." Queue the AI render for retry on next session.

FAILURE 03 · USER LOSES TRUST IN THE INTERPRETATION

What happens: After two or three corrections, the user is frustrated. Recovery: Offer the manual-build path — a panel of domain primitives that the user can drag onto a canvas and assemble directly, bypassing AI interpretation entirely. The user still gets the rendering, refinement, and export benefits. This is the escape hatch that ensures SketchForge always produces something usable, even when the AI is wrong.

§ 07What "good" looks like

The flows above are designed against measurable targets. These are the numbers we use to judge whether the system is working.

TARGET METRICS · MVP

Time from sketch start to first render: under 60 seconds median.
Interpretation accuracy on first attempt: 75% of users confirm without any corrections.
Effective accuracy after correction: 99% of confirmed specs render correctly.
Iteration completion rate: 90% of users who request an adjustment receive a satisfactory result within two iterations.
Session completion rate: 70% of started sketches reach the export stage.

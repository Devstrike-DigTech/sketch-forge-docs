§ 01System architecture at a glance

SketchForge is a mobile-first, cloud-augmented application. The phone handles capture and lightweight processing; the cloud handles semantic interpretation, generation, and persistence. The two halves communicate over a thin, well-defined API.

§ 02The eight technical layers

Each layer has a single responsibility, a defined tech choice, and a defined alternative if the primary doesn't work out.

LAYER01CAPTURE

Sketch & voice capture

Flutter · CustomPainter · GestureDetector · speech_to_text

A custom canvas widget records strokes as List<Stroke> where each stroke is {points: List<Offset>, timestamps: List<int>, pressure: List<double>}. Voice runs in parallel via the platform's native speech APIs (no audio leaves the device). The canvas exports both the structured strokes and a rasterized PNG.

LAYER02ON-DEVICE AI

Geometric primitive detection

opencv_dart · Dart FFI · ~3MB native binary per platform

A bundled OpenCV instance runs findContours() + approxPolyDP() for rectangles, triangles, and polygons, and Hough Circle Transform for circles. Shape Context distance is used for matching when primitives are ambiguous. Processing time on a mid-range phone is ~150ms for a typical sketch.

LAYER03STATE

Local state & session machine

Riverpod 2.x · Freezed for sealed unions · drift (SQLite ORM)

A finite state machine governs the session (IDLE → CANVAS_ACTIVE → DETECTING → INTERPRETING → AWAITING_CONFIRMATION → REFINING → GENERATING → RESULT). The active AssemblySpec is the source of truth; all UI reflects it reactively. Projects persist to SQLite for offline access and history.

LAYER04NETWORK

Client–server communication

Dio (HTTP) · web_socket_channel (WSS) · MessagePack for binary payloads

Sketch PNGs upload via multipart HTTPS. Interpretation results and generation progress stream over WebSocket so the user sees the spec appear progressively. Auth tokens via Bearer header (Clerk JWT). MessagePack used for sketch+stroke payloads to keep size 30-40% below JSON.

LAYER05GATEWAY

Edge API gateway

Cloudflare Workers · Hono router · rate limiting + DDoS protection

All client traffic terminates at the edge. Workers validate auth, enforce per-user rate limits, route to the appropriate backend region, and cache idempotent reads (e.g., domain vocabulary fetches). This layer absorbs malformed requests before they reach the expensive backend.

LAYER06BACKEND

API service

Node.js 22 · TypeScript · Fastify · zod for validation · Prisma ORM

Stateless HTTP service handling four main endpoints: /interpret (sketch → spec), /generate (spec → render job), /projects (CRUD), /export (asset formatting). Horizontally scaled behind the gateway. Deployed to Fly.io or Railway for region-pinned, low-latency response.

LAYER07DATA

Persistence & queue

Postgres (Supabase or Neon) · Redis (Upstash) · Cloudflare R2 for objects

Postgres stores users, projects, assembly specs, and version history. Redis handles BullMQ job queues for generation work and short-term caches. R2 holds sketch PNGs, rendered outputs, and exported assets. R2 is chosen over S3 for zero egress fees, which matters when users share renders.

LAYER08EXTERNAL AI

Vision & generation

Anthropic Claude API · Replicate (FLUX + ControlNet) · OpenAI Whisper fallback

Claude Sonnet handles sketch interpretation with forced tool-call output against the closed vocabulary. Replicate runs the ControlNet generation jobs with sketch conditioning at scale 1.3 for tight geometric lock. The procedural render path runs on the backend itself using a headless Three.js + Puppeteer setup — no external service required.

§ 03Key technical choices, with rationale

CHOICE 01 · CLIENT FRAMEWORK

Flutter over React Native

Flutter's CustomPainter gives finer control over the canvas with predictable 60fps rendering on both platforms. The Skia backend (which Flutter uses natively) is the same engine that powers most professional drawing apps. Existing team familiarity from prior projects is a major accelerator.

If this doesn't hold: fall back to React Native with react-native-skia, which gives identical canvas semantics at the cost of a slightly more fragmented build pipeline.

CHOICE 02 · INTERPRETATION MODEL

Claude Sonnet 4.5 over GPT-4o or Gemini

Claude has the strongest structured tool-use behavior of the three frontier multimodal models in 2026, which is the single most important attribute for this use case — we need it to emit valid AssemblyPart and AssemblyRelation tool calls reliably, not free text. GPT-4o is the strongest fallback; Gemini is third-line.

If this doesn't hold: dual-model voting — call both Claude and GPT-4o on the same sketch, accept only matching interpretations, defer to user when they disagree.

CHOICE 03 · GENERATION MODEL

SDXL ControlNet via Replicate (with FLUX as Pro upgrade)

SDXL ControlNet has wider model availability, faster inference (~3-5s on a high-end GPU), and better performance-per-dollar than FLUX in 2026. For the MVP it's the default. FLUX delivers higher quality but at ~3x the time and cost, so it's reserved for Pro-tier users who pay for the upgrade.

If this doesn't hold: self-host SDXL on a GPU pool (Runpod or Modal) once volume justifies it — break-even is roughly 50,000 renders per month.

CHOICE 04 · PROCEDURAL RENDER

Server-side Three.js with headless Chromium

The procedural rendering path runs on the backend using Three.js + Puppeteer in a worker pool. This gives geometrically exact, deterministic output with sub-second turnaround. The same Three.js scene definition can be exported as GLB for client-side viewing and AR placement later.

If this doesn't hold: move the procedural render to the client (Flutter Three.js bindings exist via flutter_3d_controller) and skip the server round-trip entirely — at the cost of larger client bundles.

CHOICE 05 · BACKEND HOSTING

Fly.io for the API, Cloudflare for the edge

Fly.io provides multi-region deployment with WireGuard mesh between regions, which keeps p95 latency under 200ms for users on three continents from a small instance footprint. Cloudflare Workers at the edge absorb auth, rate limiting, and bot traffic before Fly machines spin up.

If this doesn't hold: Railway or Render for simpler single-region deployment; switch when traffic justifies multi-region.

§ 04Data model — the contract between layers

Three core types govern everything. They are versioned, persisted, and exchanged across every boundary in the system.

// The structured output of Stage 02 (primitive detection) Primitive { id: string type: "rectangle" | "circle" | "ellipse" | "line" | "arc" | "polygon" bbox: [x, y, w, h] params: { ... type-specific ... } stroke_ids: number[] // which strokes formed it confidence: float } // The structured output of Stage 03 (AI interpretation) AssemblySpec { id: uuid version: int domain: "mechanical" | "plumbing" | "structural" | "furniture" parts: AssemblyPart[] relations: AssemblyRelation[] confidence: float ambiguities: string[] source: { sketch_id, primitive_ids, transcript? } } AssemblyPart { id: string type: string // from domain vocabulary position: [x, y, z] dimensions: { ... type-specific ... } orientation: quaternion material?: string // filled at Stage 05 } AssemblyRelation { from: part_id to: part_id type: "inserted_into" | "joined_to" | "stacked_on" | "perpendicular_to" | ... params: { ... } } // The fully enriched spec going into Stage 06 (rendering) EnrichedSpec extends AssemblySpec { style: "photoreal" | "blueprint" | "cad" | "clay" context: "industrial" | "residential" | "lab" | "outdoor" render_prompt: string // assembled for AI path control_image_url: string // original sketch for ControlNet }

§ 06Infrastructure implications

The above architecture has real-world consequences for hosting, cost, scaling, and operational burden. Naming these explicitly avoids unpleasant surprises at scale.

Hosting topology

The system runs across three layers: edge (Cloudflare Workers, ~free for our volume), regional API (Fly.io machines in 3 regions: US-East, EU-West, AP-Southeast), and centralized data (Postgres in one region with read replicas if needed). External AI services have their own geography; we route to whichever region the user is closest to.

Per-request cost breakdown

A single end-to-end session (sketch → interpretation → photoreal render → export) breaks down approximately as follows in 2026 pricing:

ComponentPer-request costNotes
Edge gateway (Cloudflare Workers) $0.0001 Effectively free at this scale
Backend compute (Fly.io) $0.0008 Amortized across active machines
Claude vision call (interpretation) $0.015 ~1500 input + 500 output tokens with image
Replicate SDXL ControlNet render $0.04 ~4 seconds GPU time
R2 storage + bandwidth $0.002 Zero egress fees vs S3
Postgres + Redis (amortized) $0.001 Supabase + Upstash starter tiers
Total per session (photoreal path) ~$0.059 ~$5.90 per 100 sessions

The procedural-only path (no AI render) drops the per-session cost to roughly $0.018 because the expensive Replicate call is skipped. This is why we offer it as the default for the free tier — it lets us serve a generous free allowance without bleeding cash.

Scaling thresholds

  • 0 – 10,000 sessions/month: Single Fly.io machine per region, Supabase starter, Upstash free tier. Monthly bill under $200.
  • 10,000 – 100,000 sessions/month: Scale to 2-3 Fly.io machines per region, Supabase Pro, Upstash Pay-as-you-go. Add CDN caching for renders. Monthly bill $800 – $2,500.
  • 100,000 – 1M sessions/month: Move generation to a self-hosted GPU pool (Modal or Runpod), keep Replicate as overflow. Postgres read replicas. Monthly bill $8,000 – $25,000. Margin improves significantly because GPU cost is fixed at this volume.
  • 1M+ sessions/month: Multi-region GPU clusters, custom-fine-tuned interpretation model (saves ~40% on Claude calls), dedicated CDN contract. New cost structure requires re-architecture, but unit economics are dramatically better.

Latency budget

To meet the <60-second end-to-end target from Doc 02:

  • Sketch capture: user-driven, no budget — typically 20-40s of drawing.
  • Network upload (PNG + primitives): under 1s on 4G+, under 3s on slow connections. Aggressive payload compression matters here.
  • Claude vision call: 2-4s p95. Streaming the response helps perceived latency but doesn't change wall-clock.
  • User confirmation + refinement: user-driven, no budget — typically 5-15s.
  • SDXL ControlNet generation: 4-8s p95 on Replicate. Procedural path: under 1s.
  • Final asset delivery (R2): under 500ms with CDN caching.

Privacy & data handling

Voice transcripts are processed entirely on-device — audio never leaves the user's phone. Sketches and final renders are uploaded but stored with user-controlled retention (default: 90 days for free tier, indefinite for Pro). User content is not used for model training without explicit opt-in. The correction-feedback loop that improves interpretation accuracy is opt-in and the contributed data is anonymized at the source (no user ID, no metadata).

§ 07Risk register

RISK 01 · TECHNICAL

Interpretation accuracy plateaus below acceptable threshold

If real-world interpretation accuracy is closer to 60% than 80%, user trust collapses and the correction loop becomes unbearable rather than empowering.

Mitigation: Invest early in the manual primitive labeling fallback (mentioned in Doc 02). Ship the procedural rendering path on day one so even when interpretation is wrong, users get a usable output. Collect correction data aggressively to fine-tune a domain-specific model in Phase 03.

RISK 02 · COST

Generation costs scale faster than revenue

If users average 5+ renders per session and 80%+ use the photoreal path, per-user cost exceeds the Pro subscription price. This breaks the business model at scale.

Mitigation: Default to procedural for free tier (geometrically perfect, near-zero marginal cost). Cap free-tier AI renders at 3 per day. Move to self-hosted GPU pool at 50,000 renders/month — break-even drops dramatically. Add bulk-render discount for Enterprise tier.

RISK 03 · DEPENDENCY

Anthropic API outage or pricing change

Sole-source dependency on Claude for interpretation creates a single point of failure. A 4-hour outage during peak hours would block 100% of new sessions.

Mitigation: Implement GPT-4o as a hot fallback from day one. If Claude returns 5xx or exceeds 8s, fail over to GPT-4o automatically. Cache interpretation results aggressively — repeat sketches don't re-call. Monitor pricing quarterly and build a thin abstraction layer (already in the architecture) that allows swapping providers with a one-line config change.

RISK 04 · LATENCY

Mobile network conditions degrade UX beyond recoverable

Users on slow 3G or saturated 4G see 30+ seconds of upload time, and the experience feels broken rather than responsive.

Mitigation: Compress sketch payloads aggressively (PNG → WebP saves 30%, MessagePack for metadata saves another 30%). Show optimistic UI — start the refinement chips as soon as primitive detection finishes locally, before the cloud interpretation returns. Allow offline-first capture: sketches save locally and sync when network returns.

RISK 05 · COMPLIANCE

EU AI Act and similar regulations impose constraints on generative output

2026+ regulations may require watermarking of AI-generated content, content provenance metadata, or specific disclosures to users.

Mitigation: Embed C2PA-compliant provenance metadata in every exported asset from day one — this is increasingly standard and adds negligible cost. Add a clear "generated with AI" indicator in exports. Maintain a content audit log per user (already part of the project history model) so we can respond to data requests.

§ 08Build order (engineering perspective)

The architecture above is the target state. To get there without over-investing in any single layer before validating the next, the recommended build order is:

  • Week 1-2: Flutter canvas + stroke recording + SQLite persistence. No AI yet. Goal: prove the capture layer is rock-solid.
  • Week 3: OpenCV integration via opencv_dart. Primitive detection running locally. Goal: structured output from raw sketches.
  • Week 4-5: Backend skeleton (Fastify + Postgres + Redis), Claude vision integration with forced tool calls, vocabulary definition for mechanical domain only. Goal: end-to-end interpret call returning a valid AssemblySpec.
  • Week 6: Validation overlay UI in the Flutter app. Correction flow. Goal: 99% effective accuracy on the mechanical domain via human-in-the-loop.
  • Week 7: Procedural rendering path (server-side Three.js + Puppeteer). Three.js viewer in the app. Goal: geometrically exact renders shipping.
  • Week 8: Replicate integration for AI photoreal path with ControlNet. Refinement chip UI. Goal: dual-path rendering working end-to-end. MVP complete.
  • Week 9-12: Expand domains (plumbing, structural, furniture). Voice annotation. Iteration loop. Export formats. Pro tier billing.

§ 09Operational readiness checklist

Before going to public launch, these must be in place. They are non-negotiable.

  • Structured logging (Pino → Axiom or Datadog) at every layer with request tracing.
  • Error tracking (Sentry) on both client and server, with source maps deployed.
  • Synthetic monitoring of the full pipeline (sketch upload → interpret → render → download) every 5 minutes.
  • Rate limiting at the edge (Cloudflare) and at the application layer (Fastify).
  • Cost alerts on every external service (Anthropic, Replicate, Supabase) — anomaly detection on daily spend.
  • Backup and point-in-time recovery for Postgres (Supabase handles this; verify retention).
  • Incident runbook covering: AI provider outage, R2 outage, Postgres failover, edge worker failure.
  • Privacy policy and terms of service that accurately describe data handling, including AI provider sub-processors.
  • App store compliance: privacy nutrition labels (iOS), data safety section (Android), AI-generated content disclosure where required.
